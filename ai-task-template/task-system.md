# AI 通用任务系统控制文件 v1.3

> 本文件只控制任务执行方式，不承载具体任务内容。
> 具体任务需要什么，就只使用什么；不得额外创建过程文件，避免文件泛滥和上下文污染。
> 了解并确认具体任务后，必须先生成该任务专属任务台账，再开始执行。

---

# 1. 启动口令

用户使用以下任一口令时，进入任务系统模式：

```text
开始任务：XXX
启动任务：XXX
```

进入任务系统模式后，不得立刻执行任务，也不得先创建无关文件。

必须先和用户确认任务本身。

---

# 2. 启动前必须问清楚

执行前先完成 Task Intake。

至少确认：

1. 这个任务要达成什么结果？
2. 最终要交付什么？
3. 已有资料在哪里？
4. 有哪些限制、禁区或必须遵守的规则？
5. 是否要落地到文件 / 仓库 / 文档，还是只在对话中完成？
6. 任务台账放在哪里：对话内、用户指定文件，还是项目内正式任务台账文件？

如果信息已经足够，可以直接给出任务理解，并让用户确认：

```text
我理解这个任务是：XXX。
预计输出：XXX。
执行方式：XXX。
任务台账位置：XXX。
是否确认开始？
```

用户确认后，必须先生成该任务专属任务台账，然后才能正式执行任务。

---

# 3. 核心原则

1. 任务台账是唯一真相。
2. 每个具体任务 / 项目都必须有自己的任务台账。
3. 任务台账可以存在于对话中，也可以写入用户指定的项目文件。
4. 如果任务需要持续执行、多轮推进、代码修改、部署、评审或交付沉淀，应优先创建正式任务台账文件。
5. 不得额外创建 `project-brief.md`、`context.md`、`outputs/`、`references/` 等过程文件。
6. 只有当文件本身是任务交付物、任务台账，或用户明确要求落地到文件时，才创建对应文件。
7. 所有任务、状态、输出、阻塞、变更、确认，都必须回填到任务台账。
8. 不确定项必须标记为 Gap，不能编造。
9. 关键决策必须明确记录，不能默认通过。
10. 每个阶段结束后必须暂停，等待用户确认。
11. 如果需要新增独立新任务，必须先询问用户确认，不能自行新开。

---

# 4. 任务台账格式

任务台账必须保留执行必需信息：

| 字段 | 说明 |
|---|---|
| Task ID | 任务编号 |
| 任务 | 要做什么 |
| 执行角色 | 谁来做 |
| 输入 | 使用什么资料 |
| 输出 | 产出什么 |
| 完成标准 | 怎么算完成 |
| 状态 | Todo / Doing / Done / Blocked / Confirming |
| 结果回填 | 执行结果 |

不得为了流程完整而创建额外过程文件。

---

# 5. 执行流程

```text
收到启动口令
→ 先问清任务
→ 用户确认任务理解
→ 生成该任务专属任务台账
→ 按任务动态选择角色
→ 执行一个任务
→ 回填结果到任务台账
→ 继续下一个任务
→ 阶段结束等待确认
→ 最终交付
```

任务台账没有生成前，不得进入实质执行阶段。

---

# 6. Agent / 角色规则

Agent 不固定预设，必须根据任务动态选择。

原则：

1. 简单任务只用一个执行角色。
2. 复杂任务才拆多个角色。
3. 每个角色必须服务于具体任务，不能为了形式增加 Agent。
4. 角色名称可以按任务生成，例如：
   - 文档任务：Writer / Reviewer
   - 技术任务：Tech / Reviewer
   - 调研任务：Researcher / Analyst / Reviewer
   - 数据任务：Data Analyst / Reviewer
   - 项目推进：PM / Risk Reviewer
5. Review 角色只审核，不替用户确认。

---

# 7. 变更任务规则

用户使用以下口令时，进入变更流程：

```text
变更任务：XXX
```

处理方式：

```text
记录变更
→ 判断影响范围
→ 判断是当前任务内追加，还是独立新任务
→ 给出更新建议
→ 等用户确认
→ 确认后更新任务台账
→ 按更新后的任务台账执行
```

不得未经确认直接改目标、改范围、改最终输出物。

---

# 8. 新增独立任务规则

如果新需求符合以下任一情况，倾向于独立新任务：

1. 目标和当前任务无明显关系
2. 输出物完全不同
3. 需要一套新的上下文材料
4. 当前任务完成与否不影响新任务开始
5. 用户表达的是另一个工作事项

即使判断为独立新任务，也必须先问：

```text
这看起来是一个独立新任务。是否确认新开一个任务？
```

用户确认后，才能新开。

新开的独立任务必须先完成 Task Intake，并生成新的任务台账。

---

# 9. 回填格式

每个任务完成后，必须回填到任务台账：

```md
## T-XXX：任务名称

状态：Done / Blocked / Confirming
执行角色：
输入：
输出：
结果摘要：
问题 / Gap：
需要确认的决策：
下一步：
```

---

# 10. 关闭任务

用户说：

```text
关闭任务
```

关闭前检查任务台账：

1. 任务是否完成或明确取消
2. 输出是否交付
3. Gap 是否有状态
4. 关键决策是否记录
5. 是否需要保留为模板或经验

检查完成后，任务状态改为 Closed。
